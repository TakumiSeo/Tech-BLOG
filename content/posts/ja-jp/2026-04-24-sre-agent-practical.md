# 1. SRE Agent の立ち位置

## 1.1 前提：SRE Agent は何者か

Microsoft Learn の定義を要約すると、SRE Agent は **「動いている Azure 環境に対して、人の介入を最小化しつつ運用タスクを実行する AI エージェント」** である[^1]。

特に押さえるべき特性：

- **稼働中リソース前提**：Azure CLI / REST API 経由で既存リソースを管理することが主眼。新規の設計・生成を目的とした製品ではない。
- **学習する運用知識**：調査のたびにチーム固有のパターン・根本原因・手順を記憶し、制度的知識として蓄積する（"never-leaving knowledge"）[^1]。
- **拡張点**：カスタム Runbook、サブエージェント、外部統合（Azure Monitor / Log Analytics / App Insights / Grafana / PagerDuty / ServiceNow / GitHub / ADO / Kusto / MCP）[^1]。

この特性から導かれる結論は次の一点に尽きる：

> **SRE Agent の主戦場は「環境が動き始めたあと」。作る段階では力を発揮しにくい。**

## 1.2 環境ライフサイクル × SRE Agent の相性表

時系列で整理すると、SRE Agent の相性は次のようになる。

| フェーズ | 代表的なタスク | SRE Agent 相性 | 主たる推奨ツール |
| --- | --- | :---: | --- |
| ① 環境払い出し**前**（設計・IaC 作成） | 要件定義、アーキテクチャ設計、依頼申請書 → IaC テンプレート化、命名規則、セキュリティ設計 | △ | GitHub Copilot / Copilot in Azure、Bicep / Terraform、Azure Policy |
| ② 環境払い出し**後**（障害調査・日次運用） | メトリクス異常の一次切り分け、インシデント トリアージ、影響範囲特定、ランブック実行、復旧 | **◎** | **Azure SRE Agent** |
| ③ 定期運用タスク（点検・棚卸し・レポート） | シークレット / 証明書期限抽出、月次レポート、リリース情報と構成の突合、タグ棚卸し | ○ | Azure SRE Agent（スケジュールド タスク） |

相性評価の根拠：

- **② が「◎」である理由**：稼働中リソースに対する診断・オペレーション、インシデント管理プラットフォームとの統合、MTTR 短縮という SRE Agent の設計思想そのものに合致する[^1]。
- **③ が「○」である理由**：スケジュールド タスクとして定期実行できる[^1] 一方、即応性より定型性が主で、既存の Azure Automation / Logic Apps / Functions でも代替可能な領域が含まれるため、必ずしも SRE Agent でなければならないとは限らない。ただし、結果を「調査履歴」として蓄積できる点で SRE Agent を採用する意義がある。
- **① が「△」である理由**：SRE Agent は Azure CLI / REST API を通じて**既存リソース**を操作する前提で、IaC コード生成・レビューは守備範囲外。依頼申請書 → IaC テンプレ生成のようなユースケースは、GitHub Copilot / Copilot in Azure が本命。

# 2. GitHub Copilot 系ツールとの使い分け

お客様からよくいただく「この作業は SRE Agent でやるべきか、Copilot でやるべきか？」に回答するための整理。

| 観点 | GitHub Copilot / Copilot in Azure | Azure SRE Agent |
| --- | --- | --- |
| 対象フェーズ | 設計・実装・変更（"作る"） | 稼働後の運用（"運用する"） |
| 主な対象者 | 開発者、プラットフォームエンジニア | SRE、運用チーム、オンコール |
| 代表タスク | IaC 生成、コードレビュー、ドキュメント作成、依頼申請書からの雛形化 | インシデント トリアージ、定期点検、ランブック実行、構成の影響分析 |
| インターフェース | IDE / ChatOps | Azure Portal のエージェント チャット、スケジュールド タスク、インシデント連携 |
| 学習対象 | コードベース、プロジェクト文脈 | 環境トポロジ、過去インシデント、チーム手順[^1] |
| 実行権限 | コード生成が主（実行は人が決定） | Azure リソースに対する実行アクションを持つ（要ガバナンス設計） |

**使い分けの原則（1 行で）：**

- **「作る／変える」なら GitHub Copilot**
- **「判断する／実行する／記録する」なら SRE Agent**
- **両者は補完関係**：Copilot で書いた IaC で作った環境を SRE Agent が運用する、という流れが自然。

# 3. 運用業務一覧への適用マッピング

お客様からヒアリングいただいた運用タスク例を、上記の整理にあてはめると以下のとおり。

| # | 運用タスク | フェーズ | 第一候補 | SRE Agent 活用の具体像 |
| --- | --- | --- | --- | --- |
| T1 | レポート調査中のメトリクス スパイク等について、関係各所にヒアリングする**前**に影響有無を AI に相談 | ② | **SRE Agent** | Azure Monitor / Log Analytics / App Insights を横串で参照し、対象リソース・時間帯の同時期イベント（デプロイ、スケール、依存サービス異常）を自動収集。**ヒアリング前の一次判断**を提供[^1]。 |
| T2 | 依頼申請書から IaC テンプレ生成・自動適用 | ① | **GitHub Copilot** | SRE Agent の守備範囲外。Copilot で IaC 生成 → パイプライン（GitHub Actions / ADO）で適用。SRE Agent は「適用**後**」の監視・棚卸しを担当。 |
| T3 | 最新リリース情報 URL を指示し、Azure 構成と突合してリスク・コスト最適化を AI に助言させる | ②〜③ | **SRE Agent（構成側）+ Copilot（推奨パッチ作成）** | SRE Agent がテナント内のリソース構成を取得し、指定された変更情報と突合して影響範囲を提示。改修 IaC が必要になった時点で Copilot に引き継ぐ。 |
| T4 | 障害調査・対応 | ② | **SRE Agent** | 次回アジェンダで詳細化（トリアージ、ランブック、MTTR 短縮、ポストモーテム連携）。 |
| T5 | シークレット・証明書の期限抽出、期限の近いものを指摘 | ③ | **SRE Agent（スケジュールド タスク）** | カスタム Runbook で Key Vault / App Gateway / App Service Certificate を走査し、しきい値（例：30 日以内）にかかるものを週次で通知。結果を調査履歴として蓄積。 |

## 3.1 活用パターンの類型化

運用手順書に含まれるタスクを SRE Agent に寄せる際、次の 3 類型に分類して検討すると議論が進めやすい。

1. **「一次判断」型**（T1, T3 の構成突合）
   - 目的：関係各所への問い合わせ前に、影響有無・原因仮説を得る。
   - 実装：SRE Agent への自然言語問い合わせ + Azure Monitor / App Insights 統合。
   - 効果：無駄なエスカレーションの削減、初動の高速化。

2. **「スケジュールド点検」型**（T5、T3 の定期レビュー）
   - 目的：人が忘れがちな定期確認を自動化し、抜け漏れを防ぐ。
   - 実装：スケジュールド タスク + カスタム Runbook + 通知連携。
   - 効果：証明書切れ・設定ドリフトの早期検知。

3. **「インシデント駆動」型**（T4）
   - 目的：アラート発報と同時に、トリアージ・緩和・復旧を半自動化。
   - 実装：Azure Monitor アラート / PagerDuty / ServiceNow から SRE Agent を起動、カスタム Runbook で切り分け[^1]。
   - 効果：MTTR の短縮、夜間オンコール負荷の軽減。
   - 詳細は**アジェンダ (2)「障害調査・復旧に向けた SRE Agent 活用」（次回）** にて深掘り。

## 3.2 PoC 推進にあたっての留意点

- **言語**：SRE Agent のチャット UI は現時点で英語のみサポート[^1]。お客様運用チームとのインターフェース設計時に考慮が必要。
- **自動作成リソース**：エージェント作成時に Application Insights / Log Analytics / マネージド ID が自動で作られる[^1]。命名・タグ・サブスクリプションの事前合意が必要。
- **権限設計**：SRE Agent は実行系のアクションを持つため、最小権限の原則で開始し、Runbook ごとに許可する操作スコープを明示する。
- **リージョン / テナント**：可用性はリージョン・テナント構成に依存する[^1]ため、PoC 対象サブスクリプションで事前確認。

# 4. まとめ（本資料のスコープ）

- SRE Agent の主戦場は **② 環境払い出し後** と **③ 定期運用**。設計・IaC 生成は Copilot 側に寄せる。
- 運用タスクは **一次判断型 / スケジュールド点検型 / インシデント駆動型** の 3 類型に整理すると、SRE Agent の適用可否が明確になる。
- 「作る / 運用する」で Copilot と SRE Agent を使い分け、両者を連携させるのが実践解。

次回（アジェンダ 2）では、上記類型のうち **インシデント駆動型** を掘り下げ、具体的な切り分けフロー、ランブック設計、観測統合、ポストモーテムまでを扱う。
