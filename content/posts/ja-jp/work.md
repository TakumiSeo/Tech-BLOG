## メモ: 「仕事は好きだが、引っかかりがある」

### 1. 前提（ポジティブな事実）
- 仕事（Cloud Solution Architectとしての設計/実装/検証/提案支援）は好きで、職種・環境ともに適切だと感じています。
- お客さまの課題に対して設計や検証で価値を出すこと自体は、今もやりがいがあります。

### 2. いま起きていること（引っかかりの中身）
- 以前はプライベートでも学習に時間を投下できていましたが、最近は同じようにはできなくなっています。
- 無理に勉強時間を戻したいというより、気持ちを整理して「また楽しめる状態」に戻したいです。
- もやりは常に一定ではなく、出る日と出ない日があります。
	- SNSを見た後に比較・焦りとかが出ることがある -> LinkedIn とか開くのがきつい日がある
	- アウトプットを今は自分のノートにとどめている -> でもいつかちゃんと発信したい気持ちはある
	- 仕事がうまくいっている時でも、ふと虚無感が出ることがある

### 3. 自分なりの仮説（原因は“技術そのもの”ではない）
- 日中の思考負荷（設計判断、調整、責任のある意思決定）が増え、プライベートでの情報摂取まで同じ強度で回せなくなっている。
- SNSは「比較/煽り/断言」が強く、実務の価値（地味だが長期で効く設計・運用の品質）と尺度がズレて、焦りだけが残りやすい。
- 案件の終盤は“終わりが見えにくい/積み残しが残る”ことで、達成感の回収が弱くなりやすい。
- 技術的に課題感が少なくなってきた？前は何もかも新しかったからか

### 4. 取る予定の対策（現実的に続けられる形）
- 情報摂取を「SNS中心」から「必要時に深掘り（Pull型）」へ寄せる（タイムラインを見ない/見る時間を上限設定）。
- 案件クローズ時に15分だけ、
	- Done / Not done / Next time を書き出して“終わらせる”
	- VBD デリバリ後のアンケートに答えるのもいいが、自分なりのフォーマットで整理したいかもしれない
	- 自分が良かった判断を1行で残す（達成感の回収）
- 学習は「量」ではなく「形」に寄せる（30〜60分の小さな検証、提案テンプレ化、再利用できる成果物づくり）。

※仕事が嫌になったわけではなく、「好きなまま、長期的に健全に続けるための調整」です。

## 追記（実務に落とすための具体化）

### 5. 2週間トライアル（小さく試して調整する）

“戻す”ではなく“整える”が目的なので、まずは2週間だけ試して、合うものだけ残します。

- 平日インプットは「30〜60分×週3」まで（上限を決めて罪悪感を減らす）
	- やる内容は「読む」より「小さく作る/試す」を優先（検証メモ、テンプレ、再利用物）
- SNSは「見るなら目的がある時だけ」に変更
	- タイムラインは基本オフ（見る場合は“検索/ブックマークから”）
	- どうしても見るなら、時間を決めて終了（例: 10分だけ）
- 案件クローズ時の“終わらせる儀式”（15分）
	- Done / Not done / Next time
	- 「良かった判断を1行」だけ残す

### 6. 指標（気持ちを“観測可能”にする）

感情の波を否定せず、傾向だけ掴むために軽く記録します。

- 週1回（5分）: 今週のエネルギー（0〜10）、SNS閲覧時間、やれた“小さな成果物”の数
- もやりが出た時: きっかけを1行でメモ（比較/疲労/未完了感など）
	- 対策は“その場で”打ちすぎない（傾向が見えてから調整）

### 7. “学習”の定義を変える（量→成果物）

学習量が減ったことを問題にするより、実務に残る形へ寄せます。

- 提案の型（テンプレ）を作る
- よくある設計判断のチェックリスト化（例: 非機能/運用/セキュリティ）
- 1テーマ検証 → 3行まとめ（前提/結論/注意点）


### 8. 次回共有したいこと（2週間後の確認ポイント）

- もやりの頻度/強度が変わったか
- SNSの扱いを変えて比較・焦りが減ったか
- “終わらせる儀式”で達成感が回収できたか
- 小さな成果物が積み上がったか（テンプレ/メモ/チェックリスト）
